# K8s 管理平台运维工具扩展设计方案

## 1. 当前基线

当前系统在“运维工具”侧已经有两个明确入口：

- 日志工作台：适合做多 Pod 日志查看、实时追踪、下载和 CrashLoopBackOff 前置日志拉取。
- YAML 部署：适合做直接变更、模板部署和兜底式资源操作。

但从实际排障视角看，当前平台的核心问题不是“完全没有能力”，而是能力分散在多个资源页面里：

- 事件在 Events 资源页。
- 资源关系在拓扑页。
- Pod 异常信息散落在 Pod 列表、详情抽屉、日志工作台里。
- PVC / PV / VolumeAttachment / Snapshot 相关信息分散在多个资源页。
- RBAC 可视化能力已经有基础，但不在排障路径上。

这意味着当前平台更像“资源管理台”，还不是“问题处理台”。

如果目标是让一线运维、SRE、值班同学能在 3 到 5 分钟内完成一次初步定位，那么“运维工具”栏下面应该补的是问题导向工具，而不是再堆一批资源列表。

## 2. 设计原则

### 2.1 先给结论，再给原始数据

运维排障第一步不是看 YAML，而是先判断：

- 是应用问题。
- 是调度问题。
- 是网络问题。
- 是存储问题。
- 是权限问题。
- 是发布问题。

所以运维工具页要优先展示“异常归因提示”和“影响范围”，然后再给日志、事件、YAML、原始对象。

### 2.2 一个问题只进一个入口

用户不应该先点 Pod、再点 Events、再点 Service、再点 EndpointSlice、再点 YAML，最后才发现是 selector 写错了。

运维工具的目标，是把一类问题需要跳的 4 到 6 个页面压缩成 1 个工作台。

### 2.3 优先复用现有能力，不重复造轮子

当前系统里已经有这些可复用基础：

- 多 Pod 日志工作台。
- 资源详情抽屉体系。
- Events 列表与详情事件页。
- ResourceTopologyView。
- YAML 查看 / 编辑能力。
- Manifest Apply 记录。
- RBAC Matrix 可视化配置能力。

新增运维工具应优先做“聚合入口 + 诊断视图 + 快捷动作”，不是重写同一套资源页。

### 2.4 坚持读写分离

运维工具里会天然出现强操作，例如：

- 重启。
- 驱逐 Pod。
- Cordon / Drain 节点。
- 重新部署。
- 扩容 PVC。

这些入口必须沿用当前系统的只读 / 写权限模型，不能因为在“运维工具”下就把强动作默认暴露。

### 2.5 先判断“有没有用”，再决定做不做

判断一个运维工具值不值得做，我建议至少看 5 个标准：

- 高频：是否覆盖值班最高频的故障路径。
- 减跳转：是否能把原来 4 到 6 次页面跳转压到 1 到 2 次。
- 可复用：是否主要基于当前已有数据和现有能力，而不是先补一整套新底座。
- 可结论：是否能直接给出结论、影响范围和下一步动作，而不是只堆原始信息。
- 可控：是否能在只读权限下完成大部分定位，写动作是否能被权限和审计收口。

按这 5 个标准看，这份方案里的工具总体是有用的，但价值层次应当明确区分：

- 明确高价值：故障诊断台、服务链路诊断、节点快诊中心、存储问题诊断。
- 高价值但偏观察放大：事件时间线中心、集群巡检与风险扫描。
- 高价值但更偏专项闭环：发布健康中心、RBAC 权限验证器。
- 有价值但依赖额外底座与安全控制：网络探测工具箱、变更对比与回滚助手。

### 2.6 为避免实施歧义，必须把工具边界说清楚

这些工具的方向本身没有问题，但如果不把边界写清楚，开发阶段很容易把多个工具做成“都能看一点、但都不够聚焦”的大杂烩。建议把边界先定死：

| 工具 | 主要回答的问题 | 主要入口对象 | 首期边界 |
| --- | --- | --- | --- |
| 故障诊断台 | 单个 Pod / Workload 为什么异常 | Pod / Workload | 只做诊断聚合与跳转，不替代资源详情页 |
| 服务链路诊断 | 从 Ingress / Service 到后端 Pod 哪一段断了 | Service / Ingress | 先做配置与关联判断，不做主动拨测 |
| 节点快诊中心 | 这是节点级问题还是单对象问题 | Node | 先看影响面与状态，不扩成主机运维面板 |
| 存储问题诊断 | PVC / PV / 挂载链路哪里卡住了 | PVC / PV / Snapshot | 先做链路视图与现有快捷动作，不做存储后端专有运维 |
| 事件时间线中心 | 最近一段时间集群异常是否成片爆发 | 时间范围 / reason | 只做事件放大与下钻，不替代监控系统 |
| 发布健康中心 | 这次变更为什么没成功落地 | Workload / 部署记录 | 先做发布结果归因，不直接做自动回滚 |
| RBAC 权限验证器 | 某个主体为什么被拒绝 | User / ServiceAccount | 先做验证与补权建议，不做一键授权 |
| 集群巡检与风险扫描 | 当前有哪些未爆炸但高风险的问题 | 规则包 / 集群 | 先做规则命中与跳转，不做自动修复 |
| 网络探测工具箱 | 从某个源到某个目标是否通 | 源 Pod + 目标 | 先做受控探测，不开放任意命令执行 |
| 变更对比与回滚助手 | 这次改动到底改坏了什么 | Workload / Apply 记录 | 首期先做 diff 与回滚清单，不直接做一键回滚 |

### 2.7 设计修正：先做“诊断闭环”，再做“执行闭环”

基于上面的价值和边界判断，这份方案在落地顺序上还应再收紧两条：

- 第一阶段优先做读诊断闭环，也就是先把“结论、关联、证据、跳转”做完整，让值班同学能快速定位。
- 第二阶段再做执行闭环，也就是在已有权限、审计和回放保护的前提下，再开放回滚、探测、批量修复这类强动作。

这样做的好处是：第一阶段就能明显提升排障效率，同时避免平台过早进入“动作很多，但风险控制不够”的状态。

## 3. 推荐新增工具清单

### 3.1 P0：故障诊断台

#### 目标场景

- 某个 Pod 一直重启。
- 某个工作负载 ready 不起来。
- 某次告警只给了 Pod 名称或 Deployment 名称。

#### 页面设计

输入区：

- 集群。
- 命名空间。
- 资源名称关键字。
- 资源类型切换：Pod / Deployment / StatefulSet / DaemonSet / Job。

诊断主区建议拆成 6 张卡：

1. 诊断结论卡：直接给出“最可能原因 Top 3”。
2. Pod 运行态卡：Phase、Ready、Restart、Node、容器状态、退出码、最近一次重启时间。
3. 事件时间线卡：按时间聚合 Warning / Normal 事件。
4. 日志入口卡：当前日志、前次崩溃日志、异常容器高亮。
5. 依赖关系卡：所属工作负载、对应 Service、PVC、Node。
6. 快捷动作卡：查看 YAML、打开详情、重启工作负载、进入日志工作台。

#### 为什么要做

这是最高频的值班场景。现在平台虽然有日志、详情、事件，但入口是资源导向的，不是故障导向的。故障诊断台能把“看日志 + 看事件 + 看容器状态 + 看依赖”收拢成一个页面，值班效率会明显提升。

#### 可复用现有能力

- 复用 MultiPodLogWorkbench。
- 复用 Pod / Workload 详情抽屉。
- 复用现有 warning event 统计。
- 复用 YAML 查看能力。

### 3.2 P0：服务链路诊断

#### 目标场景

- Service 没流量。
- Ingress 已创建但访问不通。
- Service 有 selector，但没有后端 Pod。
- EndpointSlice 存在但 readiness 不正常。

#### 页面设计

输入区：

- 命名空间。
- Service / Ingress 名称。

主视图区建议按链路顺序展示：

1. Ingress / Gateway 入口卡。
2. Service 配置卡：type、clusterIP、ports、selector。
3. Endpoints / EndpointSlice 卡：ready / notReady、端口、目标 Pod。
4. Pod 后端卡：后端 Pod 数量、ready 数、所在 Node。
5. NetworkPolicy 影响卡：是否可能拦截。
6. 结论卡：
   - selector 不匹配
   - 无后端 Pod
   - 后端 Pod 不 ready
   - 端口映射异常
   - Ingress 后端服务名不匹配

#### 为什么要做

这类问题在现网里非常常见，而且最容易在 Service、Endpoints、EndpointSlice、Pod、Ingress 之间来回跳。当前系统已经补了 Endpoints / EndpointSlice 详情抽屉，但还缺少“链路级诊断视角”。

#### 可复用现有能力

- 复用 ResourceTopologyView 的资源关联关系。
- 复用 Endpoints / EndpointSlice 详情抽屉。
- 复用 Services / Ingresses / NetworkPolicies 列表与 YAML 能力。

### 3.3 P0：节点快诊中心

#### 目标场景

- 某个节点 NotReady。
- 节点磁盘、内存、PID 压力导致 Pod 异常。
- 某个节点上的 Pod 集体重启或漂移。

#### 页面设计

建议围绕“节点影响面”设计，而不是只展示 Node 对象：

1. 节点状态卡：Ready、Kernel、ContainerRuntime、版本、污点、Unschedulable。
2. 压力信号卡：MemoryPressure、DiskPressure、PIDPressure、NetworkUnavailable。
3. 受影响 Pod 卡：该节点上的异常 Pod、重启次数高的 Pod、Pending Pod。
4. 受影响工作负载卡：Deployment / StatefulSet / DaemonSet 维度汇总。
5. 事件卡：最近 Warning 事件。
6. 快捷动作卡：查看节点 YAML、查看节点相关 Pod、Cordon / Uncordon / Drain。

#### 为什么要做

节点问题是典型“影响范围大、恢复时间长”的故障源。现在平台如果要看节点影响面，操作链太长。节点快诊中心应该让值班同学先判断“这是单 Pod 问题还是节点级问题”。

#### 可复用现有能力

- 复用 Nodes 列表与 YAML 能力。
- 复用 Pod 警告事件统计。
- 复用工作负载与 Pod 列表筛选能力。

### 3.4 P0：存储问题诊断

#### 目标场景

- PVC Pending。
- PV 已绑定但 Pod 挂载失败。
- StorageClass 参数导致 provisioning 异常。
- VolumeAttachment 异常。
- Snapshot 创建成功但恢复链路异常。

#### 页面设计

建议以 PVC 名称为主入口，也支持 PV / VolumeSnapshot 反查：

1. 绑定关系卡：PVC -> PV -> StorageClass。
2. 挂载关系卡：PVC -> Pod -> Node -> VolumeAttachment。
3. 容量状态卡：request / capacity / expansion 状态。
4. Snapshot 状态卡：snapshot class、readyToUse、bound content、restore size。
5. 事件卡：ProvisioningFailed、AttachVolume、MountVolume 等关键事件。
6. 快捷动作卡：扩容 PVC、编辑 PV、查看 YAML、打开相关 Pod 日志。

#### 为什么要做

存储问题一般跨多个资源对象，单看 PVC 是定位不出来的。当前系统已经有 PVC 扩容、PV 编辑、VolumeSnapshot 详情，但还没有把这条链路串成一个运维工作台。

#### 可复用现有能力

- 复用 PVC 扩容弹窗。
- 复用 PV 编辑弹窗。
- 复用 VolumeSnapshot 详情抽屉。
- 复用 StorageClass / PV / PVC / VolumeAttachment YAML 能力。

### 3.5 P1：事件时间线中心

#### 目标场景

- 最近 10 分钟集群里到底发生了什么。
- 是否有同类 Warning 在多个命名空间同时出现。
- 哪些异常是瞬时噪音，哪些是持续性问题。

#### 页面设计

建议不是再做一个普通 Events 列表，而是做“运维态聚合”：

1. 时间轴视图：按分钟聚合事件峰值。
2. Top Reason 榜单：BackOff、FailedMount、FailedScheduling、ImagePullBackOff 等。
3. 影响对象榜单：受影响 Pod / Workload / Node 数量。
4. 事件下钻：点击 reason 后自动关联到对应资源。
5. 规则高亮：持续出现超过阈值的 reason 直接标红。

#### 为什么要做

Events 资源页偏“原始数据浏览”，而运维排障需要的是“异常放大镜”。事件时间线中心的价值，在于让用户先看到趋势和爆点，再决定下钻到哪个对象。

#### 可复用现有能力

- 复用现有 EventsPanel。
- 复用 dashboardEvents。
- 复用资源详情抽屉里的事件加载逻辑。

### 3.6 P1：发布健康中心

#### 目标场景

- Deployment 发布后 ready 不上来。
- StatefulSet 滚动更新卡住。
- HPA / PDB / 镜像拉取 / 探针失败影响发布结果。

#### 页面设计

1. 发布对象列表：最近变更的 Deployment / StatefulSet / DaemonSet / Job。
2. 健康结论卡：成功、进行中、阻塞、回退风险。
3. 阻塞原因卡：
   - 镜像拉取失败
   - 探针失败
   - PDB 限制
   - 调度失败
   - 节点资源不足
4. 变更对比卡：当前镜像、前一版本镜像、最近一次 YAML 部署记录。
5. 快捷动作：重启、回到 YAML、打开日志工作台、看事件。

#### 为什么要做

发布失败是平台最常见的线上操作风险之一。当前平台具备 YAML 部署和 Manifest Apply 记录，但缺一个“发布结果复盘页”。这个工具能把“变更动作”和“运行结果”串起来。

#### 可复用现有能力

- 复用 Manifest Apply 记录。
- 复用工作负载详情与事件。
- 复用日志工作台。

### 3.7 P1：RBAC 权限验证器

#### 目标场景

- 某个 ServiceAccount 为什么不能读 Pod 日志。
- 某个用户为什么看得到列表但不能执行写操作。
- 新建 Role / RoleBinding 后，权限是否真的生效。

#### 页面设计

1. 主体输入：User / ServiceAccount。
2. 目标输入：namespace、resource、verb。
3. 验证结果卡：允许 / 拒绝。
4. 规则来源卡：命中了哪个 Role / ClusterRole / Binding。
5. 最小补权建议卡：如果拒绝，建议补哪条规则。
6. YAML 输出卡：生成最小补权建议 YAML。

#### 为什么要做

权限问题本质上也是排障问题，而且往往很耗时。当前平台已有 RBACMatrixDrawer，这是非常好的基础，但还缺少“验证式入口”。验证器能把“设计权限”转成“验证权限”。

#### 可复用现有能力

- 复用 RBACMatrixDrawer 的权限模型。
- 复用 Role / Binding / ClusterRole / ClusterRoleBinding 编辑与 YAML 能力。

### 3.8 P1：集群巡检与风险扫描

#### 目标场景

- 每天例行巡检。
- 变更前做一次快速体检。
- 值班交接时快速看当前风险面。

#### 页面设计

建议分为规则组：

1. 工作负载风险：长时间未 ready、重启过高、无 requests/limits。
2. 服务风险：Service 无 endpoints、Ingress 后端无服务。
3. 存储风险：PVC Pending、PV Released 未清理、Snapshot 异常。
4. 节点风险：NotReady、Pressure、污点异常。
5. 安全风险：高权限 ClusterRoleBinding、默认命名空间暴露、匿名访问风险。
6. 配额风险：ResourceQuota 高水位、LimitRange 缺失。

每条规则都要支持：

- 风险等级。
- 命中对象数。
- 一键跳转到对象列表。

#### 为什么要做

巡检工具的价值不在“发现新问题”，而在“把未爆炸的问题提前显性化”。这类工具能显著降低值班被动救火概率。

#### 可复用现有能力

- 复用现有资源列表接口。
- 复用 ResourceQuota / LimitRange / RBAC / Storage / Node 等现有面板数据。

### 3.9 P2：网络探测工具箱

#### 目标场景

- DNS 解析是否正常。
- Service 端口是否通。
- Pod 到 Pod / Pod 到 Service / Pod 到域名连通性是否正常。

#### 页面设计

1. 选择源 Pod。
2. 目标输入：Pod IP / Service / 域名 / 端口。
3. 探测方式：DNS、TCP、HTTP、HTTPS。
4. 结果区：解析结果、连接延时、HTTP 状态、错误摘要。
5. 探测历史区：最近 N 次结果。

#### 为什么要做

这类工具非常适合线上快速排障，但它依赖后端提供受控的探测执行能力，安全边界要设计好，所以优先级可以放在 P2。

#### 首期边界

- 首期只支持 DNS、TCP、HTTP / HTTPS 这类低副作用探测。
- 目标范围限制在集群内资源和白名单域名，不开放任意外部目标。
- 所有探测动作必须记录审计日志，并限制超时、并发和请求频率。

#### 可复用现有能力

- 复用现有 Pod Exec / 会话通道能力，封装成受控探测动作。
- 复用 Service、Pod、EndpointSlice 等现有资源数据。
- 复用审计日志体系记录探测来源和参数。

### 3.10 P2：变更对比与回滚助手

#### 目标场景

- 一次 YAML 部署后服务异常。
- 想确认到底改了哪些字段。
- 需要快速回滚到上一次成功版本。

#### 页面设计

1. 变更对象选择。
2. 当前 YAML vs 上次部署 YAML diff。
3. 关键字段高亮：镜像、env、resources、selector、ports、storageClass。
4. 首期生成回滚清单，后续再支持一键恢复到上一次成功版本。
5. 关联审计日志与部署记录。

#### 为什么要做

很多线上故障都不是“资源坏了”，而是“刚改坏了”。这个工具的价值是把平台从“可操作”提升到“可回放、可回退”。

#### 首期边界

- 首期先做 YAML diff、关键字段高亮和回滚清单生成。
- 一键回滚放到后续阶段，前提是补齐成功版本标记、审批链路和审计保护。
- 回滚对象首期限定在 Deployment、StatefulSet 这类幂等型工作负载，不直接覆盖所有资源。

#### 可复用现有能力

- 复用 Manifest Apply 记录。
- 复用 YAML 查看 / 编辑能力。
- 复用审计日志与工作负载详情能力。

## 4. 运维工具栏建议信息架构

建议不要把运维工具栏做成无序列表，而是按排障思路组织：

### 4.1 快速排障

- 故障诊断台
- 服务链路诊断
- 节点快诊中心
- 存储问题诊断
- 事件时间线中心

### 4.2 发布与变更

- YAML 部署（现有）
- 发布健康中心
- 变更对比与回滚助手

### 4.3 权限与安全

- RBAC 权限验证器

### 4.4 巡检体检

- 集群巡检与风险扫描

### 4.5 观察分析

- 日志工作台（现有）
- 网络探测工具箱

## 5. 推荐落地顺序

如果只按“投入产出比”排，我建议是：

1. 故障诊断台。
2. 服务链路诊断。
3. 节点快诊中心。
4. 存储问题诊断。
5. 事件时间线中心。
6. RBAC 权限验证器。
7. 发布健康中心。
8. 集群巡检与风险扫描。
9. 网络探测工具箱。
10. 变更对比与回滚助手。

原因很直接：

- 前 4 个解决的是最常见、最耗时、最容易来回跳页面的问题。
- 第 5 到第 8 个解决的是“运维效率”和“值班质量”。
- 最后两个更强，但依赖更多后端控制、审计与安全边界设计。

## 6. 结论

在当前系统下，“运维工具”最缺的不是再加几个资源入口，而是补齐 3 类能力：

1. 问题导向的聚合诊断页。
2. 资源链路级的关联分析页。
3. 变更、权限、风险的运维闭环页。

如果只允许先做一批，我建议优先落地这 5 个：

- 故障诊断台
- 服务链路诊断
- 节点快诊中心
- 存储问题诊断
- 事件时间线中心

这一批做完，平台就会从“能管理资源”明显提升到“能真正拿来排障”。
